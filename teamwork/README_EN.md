# door_sensor

This is an `STM32F103` door-state sensing example project. The current codebase mainly provides the following features:

- Distance measurement using the `VL6180X` sensor
- Sensor communication through `I2C1 + DMA`
- Status indication through multiple user `LED`s
- Periodic measurement, key-mode switching, and alarm control inside the `SysTick` interrupt

## Recommended Reading Order

If this is your first time working with the project, this order is recommended:

1. `Core/Src/main.c`
2. `Core/Inc/bsp.h`
3. `Core/Src/bsp.c`
4. `User_Drives/vl6180x.h`
5. `User_Drives/Src/vl6180x.c`
6. `User_Drives/user_led.h`
7. `User_Drives/Src/user_led.c`
8. `Core/Src/stm32f1xx_it.c`

## File Responsibilities

### 1. `Core/Inc/bsp.h`

This file centrally declares shared cross-file objects such as:

- `user_led[6]`
- `red_led`
- `meas`
- `hi2c1`
- `user_time_mode`

When a new module is added, its `extern` declarations are usually added here first.

### 2. `Core/Src/bsp.c`

This file defines the global objects declared in `bsp.h` and applies zero initialization.

You can think of it as:

- Module instance storage
- Global runtime state storage
- Cross-module callback hookup point

### 3. `Core/Src/main.c`

This is the main program entry point. It mainly:

- Calls `HAL_Init()`
- Configures the system clock
- Initializes `GPIO / DMA / I2C1`
- Calls `LED_Init()` to initialize the LED module
- Calls `VL6180_Init()` to initialize the ranging sensor

Note:

- The “initialization” in `bsp.c` is closer to zero-filling object storage
- The initialization in `main.c` is the real hardware and module bring-up

### 4. `Core/Src/stm32f1xx_it.c`

This is one of the most important runtime logic files in the project. In addition to interrupt entries, it also handles:

- `SysTick` 1 ms software time base maintenance
- Periodic triggering of `VL6180X` measurements
- Long-press based mode switching
- Alarm LED control
- Forwarding of `I2C`/`DMA` completion callbacks

If the program shows symptoms such as “measurement not updating”, “key mode behaving incorrectly”, or “unexpected LED behavior”, check this file first.

### 5. `User_Drives/Src/vl6180x.c`

This file implements the `VL6180X` sensor driver. Its main characteristics are:

- Blocking setup during initialization
- Non-blocking `I2C` access during measurement
- State-machine-driven ranging flow
- Dependence on the I2C completion callbacks in `stm32f1xx_it.c`

### 6. `User_Drives/Src/user_led.c`

This file implements the LED driver. It is responsible for:

- Storing the GPIO port and pin used by each LED
- Supporting both active-high and active-low wiring
- Providing `On / Off / Toggle / Get_State` interfaces

## Current Feature Overview

From the current code, the project already implements the following basic features:

- Initialization of 1 red LED and 6 user LEDs
- Initialization of the `VL6180X` ranging module
- Fixed-period distance measurement triggering
- Alarm control based on measured distance and thresholds
- Mode switching among normal use mode, time-setting mode, and distance-setting mode

## Development Notes

- Daily changes should usually stay in `Core` and `User_Drives`
- Files under `Drivers` are third-party libraries and normally should not be edited directly
- If `door_sensor.ioc` is modified, re-check whether CubeMX regeneration overwrote any custom code
- Custom code should preferably stay inside `USER CODE BEGIN/END` regions

## Build Notes

This project already contains `CMakeLists.txt` and `CMakePresets.json`, so it can be organized with CMake. It also keeps the `STM32CubeMX` and IDE-related project files.

If you are mainly reading or maintaining the code, you usually do not need to understand the full build scripts first. Start with `Core` and `User_Drives`.
